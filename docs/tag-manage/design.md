# Tag 管理设计文档

## 1. 目标
基于原始需求“提供一个 Tag 管理页面并支持 CRUD”，定义前端交互与业务规则，确保后续实现与测试有统一输入。

## 2. 用户故事
- 作为内容管理人员，我希望查看当前所有 Tag，以便快速了解系统标签现状。
- 作为内容管理人员，我希望新建 Tag，以便为内容分类新增可用标签。
- 作为内容管理人员，我希望编辑 Tag 信息，以便修正命名或描述。
- 作为内容管理人员，我希望删除无效 Tag，以便保持标签体系整洁。

## 3. 页面与交互范围
- 页面名称：Tag 管理
- 核心区域：
  - 顶部标题与说明
  - 搜索框（按名称筛选）
  - “新建 Tag”按钮
  - Tag 列表表格（名称、描述、创建时间、操作）
  - 新建/编辑弹窗
  - 删除确认弹窗
- 交互目标：用户在同一页面完成 Tag 的增删改查。

## 4. 字段定义
- `id`：Tag 唯一标识，只读。
- `name`：Tag 名称，必填，长度 1-30，名称唯一。
- `description`：Tag 描述，选填，长度 0-120。
- `createdAt`：创建时间，只读。
- `updatedAt`：更新时间，只读。

## 5. EARS 需求（功能）

### 5.1 Ubiquitous
- 系统应在 Tag 管理页面始终展示可见的操作反馈（成功/失败提示）。
- 系统应在所有写操作后保持列表数据与服务端一致。

### 5.2 Event-driven
- 当用户进入 Tag 管理页面时，系统应加载并展示 Tag 列表。
- 当用户点击“新建 Tag”并提交合法表单时，系统应创建 Tag 并刷新列表。
- 当用户点击“编辑”并提交合法表单时，系统应更新对应 Tag 并刷新列表。
- 当用户确认删除 Tag 时，系统应删除对应 Tag 并刷新列表。
- 当用户在搜索框输入关键字时，系统应按名称过滤列表结果。

### 5.3 State-driven
- 当页面处于“加载中”状态时，系统应禁用提交按钮并显示加载指示。
- 当页面处于“空数据”状态时，系统应显示空状态文案并引导用户新建首个 Tag。

### 5.4 Optional
- 若用户未填写描述，系统应允许提交并保存为空字符串。
- 若名称首尾包含空白字符，系统应在提交前去除首尾空白。

### 5.5 Unwanted
- 若名称为空或超长，系统应阻止提交并展示字段级错误信息。
- 若名称重复，系统应阻止提交并提示“Tag 名称已存在”。
- 若删除目标不存在，系统应提示“Tag 不存在或已删除”并刷新列表。
- 若网络请求失败，系统应保留当前输入并提示可重试。

## 6. 验收标准
- 用户可在单页完成 Tag 的创建、查询、编辑、删除。
- 表单校验覆盖必填、长度、唯一性约束。
- 删除动作有二次确认，避免误删。
- 列表、弹窗、反馈状态在桌面与移动端可用。

## 7. 非功能要求
- 可用性：常规 CRUD 操作流程不超过 3 次主交互。
- 性能：列表首屏加载应在可接受范围内（本地开发环境体感流畅）。
- 可维护性：交互文案与字段定义统一，便于后续扩展（如分页、批量操作）。
